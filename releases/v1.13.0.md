# v1.13.0

## In-project maps and overlays are now actually enforced

`--map .git` was silently ineffective: the read-only bind was emitted
before the project's read-write bind, and bwrap gives the later mount
precedence — so the agent could still write `.git` (history, hooks)
despite the user asking for read-only. Reported with an accurate
bwrap-side diagnosis by @yetisunny (#83). Auditing the report surfaced
two siblings of the same bug, one of them a data-integrity hole.

### Fixed

- **`--map` on a path inside the project directory is now read-only as
  requested** (#83). Mounts whose destination sits inside the project
  are emitted after the project bind (and before mask/deny overlays,
  which still win). Out-of-project maps are unchanged.
- **`--overlay-map` on an in-project path no longer leaks writes into
  the real files.** The same shadowing let writes bypass the
  copy-on-write layer entirely — the overlay dir was created, but
  edits mutated the original project files. In-project overlays now
  mount above the project bind, so writes land on the side layer as
  documented.
- **macOS had the same hole through SBPL's last-match-wins**: the
  project-subtree write allowance covered in-project `--map` paths.
  The profile now emits explicit `(deny file-write*)` rules for
  `ro_maps` and `overlay_maps` after the allows (also hardening the
  documented read-only overlay fallback on macOS).

### Changed

- Documented that Landlock cannot enforce read-only-inside-read-write
  (access rights are unioned, no deny semantics); the bwrap mount
  layer owns that enforcement on Linux. README now shows the
  `--map .git` pattern for protecting Git history and hooks from the
  agent, and CLAUDE.md's mount-order list gained the two in-project
  groups.

### Added

- Mount-order regression tests: in-project ro map binds after the
  project mount, out-of-project maps stay before it, mask overlays
  still stack on top, and in-project overlays mount after the project.
- macOS profile test that write denies for ro/overlay maps appear
  after the project write allowance.

### Validation

- Verified formatting, clippy (Linux and aarch64-apple-darwin
  cross-target, all targets), and the full test suite before release.
- Live end-to-end on Linux: `--map .git` is readable but returns EROFS
  on write while the project stays writable; an in-project
  `--overlay-map` write lands on the side layer with the host file
  untouched.
