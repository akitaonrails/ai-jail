# v1.15.0

## First-class ai-memory managed workstreams

This release recognizes the harness selected by `ai-memory run` while keeping
the launcher and child process inside one sandbox.

### Added

- `ai-jail ai-memory run claude` and `ai-jail ai-memory run codex` now apply
  harness-specific terminal behavior and global command configuration.
- Global `[commands.ai-memory]` settings are preserved and layered before the
  selected `[commands.<harness>]` settings. Project config and CLI flags keep
  their existing higher precedence.
- Private-home command discovery now considers both the outer ai-memory
  executable and its selected native harness. Agent state remains hidden in
  private-home unless explicitly mapped.
- Linux normal mode mounts `~/.local/share/ai-memory` read-write so native
  lifecycle hooks can maintain their local spool and session state. macOS
  already grants normal-mode write access to `~/.local`.

### Fixed

- Managed OpenCode sessions bypass the status-bar PTY proxy, matching direct
  OpenCode launches and avoiding ambiguous-width rendering corruption.
- Managed Codex sessions receive the same resize redraw key as direct Codex
  launches.

### Compatibility

- Direct commands and non-`run` ai-memory subcommands keep their previous
  behavior.
- The parser accepts ai-memory's documented wrapper options, `--option=value`
  forms, harness aliases, and explicit `--executable` paths. Unknown or
  incomplete wrapper syntax remains scoped to the outer command rather than
  guessing.

### Validation

- Added focused tests for command parsing, configuration precedence, terminal
  behavior, private-home executable discovery, and Linux ai-memory state
  mounts.
- Verified formatting, clippy with warnings denied, the complete Linux test
  suite, aarch64-apple-darwin all-target clippy, and live Linux sandbox probes
  before release.
