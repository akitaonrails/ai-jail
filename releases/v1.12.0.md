# v1.12.0

## Private-home mode can start home-installed agents

`ai-jail --private-home claude` — the README's own example — failed
with ENOENT when the agent was installed the official way, because the
installer targets `~/.local/bin` and private home hides everything
under `$HOME`, the agent binary included. Reported with an excellent
root-cause analysis by @mdindoffer (#81).

### Fixed

- **`--private-home` now resolves the invoked command on the host and,
  when its binary lives under `$HOME`, ro-binds the symlink chain and
  install directory into the jail** (#81). This is command-driven, not
  a curated agent list: any tool installed under the home directory
  works, and only the program actually being run becomes visible —
  config, cache, and state stay hidden, preserving the private-home
  contract. The same paths join the seatbelt read allowlist on macOS,
  which had the identical bug.

### Changed

- Deliberately excluded from the exemption: `--lockdown` (clears the
  environment down to the system `PATH`, so home-installed binaries
  are unreachable there by design) and browser mode (system browsers).
  Tools that need more than their install directory at startup keep
  the `--map` escape hatch, now documented in the README.

### Added

- Resolver unit tests covering the installer symlink-chain layout,
  absolute commands, system binaries (no-op), missing/relative
  commands, and symlink loops.
- bwrap wiring tests that private-home dry-run args carry the ro-binds
  and that lockdown does not.
- macOS seatbelt test that the private-home profile grants read on the
  resolved install paths.

### Validation

- Verified formatting, clippy (Linux and aarch64-apple-darwin
  cross-target, all targets), and the full test suite before release.
- Live end-to-end on Linux with an isolated `$HOME` mirroring the
  official installer layout: private-home runs the agent (previously
  ENOENT), default mode unchanged, and a sibling `$HOME` file remains
  invisible inside the jail.
